# implement

이슈의 작업 계획을 기반으로 커밋 단위별 구현을 진행한다.

## 입력

`/implement <이슈번호>` 형식으로 실행한다 (예: `/implement 1`).  
이슈 번호가 없으면 `git branch --show-current` 로 현재 브랜치를 확인한 뒤 사용자에게 이슈 번호를 묻는다.

## 실행 순서

1. **브랜치 확인**
   - `git branch --show-current` 실행
   - 이슈 작업 계획 코멘트의 `브랜치:` 항목과 현재 브랜치를 비교한다
   - 다르면 경고 후 계속 여부를 사용자에게 확인한다 (예: `현재 브랜치: dev, 예상 브랜치: feature/initial-setup`)

2. **이슈 + 작업 계획 읽기**
   - `gh issue view <N> --comments` 로 이슈 본문과 코멘트(작업 계획)를 읽는다
   - `CLAUDE.md`의 `## 상세 문서` 섹션을 확인해 관련 docs 파일을 읽는다
   - 관련 docs에서 이번 구현과 직접 연관된 스펙(타입, API, 구조 등)을 추출해 아래 형식으로 출력한다:
     ```
     구현 스펙 확인:
     - (스펙 항목 1)
     - (스펙 항목 2)
     ...
     이 스펙대로 구현합니다.
     ```
   - 구현 중 이 스펙과 다른 방향이 필요해지면 즉시 작업을 멈추고 docs를 먼저 수정한 뒤 계속 진행한다

3. **커밋 단위 파악**
   - 작업 계획 코멘트의 `### 커밋 단위` 섹션을 읽는다
   - 코멘트에 `### 커밋 단위`가 없으면 사용자에게 알리고 `/plan-sprint`를 먼저 실행할 것을 안내한다
   - `git log --oneline` 으로 이미 완료된 커밋을 확인해 남은 단위를 파악한다
   - 남은 단위 목록을 출력하고 몇 번째 단위부터 진행할지 사용자에게 확인한다
   - 단위가 하나(단일)이면 확인 없이 바로 진행한다
   - **TodoWrite로 남은 커밋 단위 체크리스트를 생성한다** — 대화가 끊겨도 진행 상태를 유지하기 위함

4. **구현**
   - 선택한 커밋 단위의 파일/작업을 구현한다
   - CLAUDE.md 규칙을 반드시 준수한다:
     - 구현 방향이 docs 설계와 다르면 코드 전에 해당 docs 파일을 먼저 수정한다
     - `any` 사용 금지 — 타입 모를 경우 `unknown` 후 narrowing
     - 함수형 컴포넌트만 사용 (class component 금지)
     - `dangerouslySetInnerHTML` 직접 사용 금지

5. **품질 검증**
   - `npm run lint` 실행
   - `npm run build` 실행
   - 오류가 있으면 즉시 수정 후 재실행한다

6. **완료 보고**
   - 구현한 파일 목록을 출력한다
   - 커밋 진행률을 표시한다 (예: `커밋 단위 1/3 완료`)
   - 해당 커밋 단위 Todo 항목을 완료 처리한다
   - `/commit-msg` 로 커밋할 것을 안내한다
   - 마지막 단위 완료 시:
     - 작업 계획의 `### 완료 기준` 체크리스트를 출력한다
     - `/docs-sync` 실행을 안내한다
