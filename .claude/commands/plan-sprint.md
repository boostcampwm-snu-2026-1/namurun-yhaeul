# plan-sprint

구현 작업을 시작하기 전에 이슈를 검토하고 작업을 계획한다.

## 실행 순서

1. `gh issue list --state open` 으로 열린 이슈 전체 목록을 확인한다.

2. 의존관계를 간략히 파악해 추천 다음 작업을 결정한 뒤, 아래 형식으로 출력하고 사용자에게 선택을 요청한다:

```
## 열린 이슈

#1 이슈명
#2 이슈명
...

추천: **#N "이슈명"** — 추천 이유 한 줄

어떤 이슈를 진행할까요? (번호로 답해주세요. 이슈를 수정/추가하고 싶으면 말씀해주세요.)
```

3. 사용자가 이슈 번호를 선택하면:
   - `CLAUDE.md`의 `## 상세 문서` 섹션을 확인해 관련 docs 파일을 읽는다.
   - 해당 이슈를 검토한다:
     - 이슈 단위가 한 PR로 머지 가능한 크기인가? 너무 크면 분리를 제안한다.
     - 빠진 작업이 있는가?

4. 검토 결과와 작업 계획을 출력한다:

```
## 작업 계획 — #N "이슈명"

브랜치: feature/xxx

### 이슈 검토
- (분리/추가 제안이 있을 경우만) 제안 내용

### 구현 범위
- 만들 파일 / 수정할 파일 목록
- 주요 구현 포인트

### 완료 기준
- 이 이슈가 "완료"됐다고 볼 수 있는 조건
```

5. 브랜치 생성 명령어를 안내한다:
   ```
   git checkout -b feature/xxx
   ```
   브랜치 생성은 직접 실행하지 않는다 — 사용자가 확인 후 실행.
